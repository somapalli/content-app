## Summary: Vendor File Intake Options

You have two main options:

```text
Option 1: Mover
Vendor / On-Prem file location → Mover → Azure Blob

Option 2: Azure Blob SFTP with HNS
Vendor → SFTP directly into Azure Blob Storage
```

## Option 1: Mover — On-Prem to Azure Blob

In this option, the vendor sends the file to an existing **on-prem file/SFTP location**, and then **Mover transfers the file from on-prem to Azure Blob Storage**.

| Pros                                                              | Cons                                                                     |
| ----------------------------------------------------------------- | ------------------------------------------------------------------------ |
| Uses existing on-prem file intake pattern                         | Still depends on on-prem infrastructure                                  |
| May avoid enabling HNS/SFTP in Azure Blob                         | More moving parts: vendor → on-prem → Azure                              |
| Easier if vendor already sends files to on-prem today             | Additional latency before file reaches Azure                             |
| Lower Azure storage architecture change                           | More operational support needed for mover jobs                           |
| Can reuse existing firewall, network, and vendor setup            | Failure points increase: on-prem server, mover job, network, blob upload |
| May be easier from approval perspective if process already exists | Not fully cloud-native                                                   |
| Good short-term or migration option                               | Harder to scale and standardize long term                                |

**Best fit:**
Use this if you already have a working on-prem file intake process and need a **lower-change, short-term path** to move files into Azure.

---

## Option 2: HNS with SFTP Enabled on Azure Blob

In this option, you create a storage account with **Hierarchical Namespace enabled**, then enable **SFTP**, allowing the vendor to send files directly to Azure Blob Storage.

| Pros                                                     | Cons                                                                |
| -------------------------------------------------------- | ------------------------------------------------------------------- |
| Cloud-native and fully managed                           | Requires HNS approval                                               |
| Vendor sends files directly to Azure                     | HNS changes the storage account behavior                            |
| Removes dependency on on-prem file servers               | Better to create a new storage account, not modify existing one     |
| Fewer hops and fewer failure points                      | Requires security review for external vendor access                 |
| Supports folder-level access and SFTP local users        | Need network controls, IP allowlist, SSH key management             |
| Easier to trigger downstream Azure processing            | SFTP feature may add cost                                           |
| Better long-term architecture                            | Existing apps may need compatibility review if using same account   |
| Good audit, monitoring, lifecycle, and event integration | Requires operational readiness: logging, alerts, retention, archive |

**Best fit:**
Use this if the goal is a **long-term Azure-native secure file intake pattern**.

---

## Simple comparison

| Criteria               | Mover Option                   | HNS + SFTP Option                           |
| ---------------------- | ------------------------------ | ------------------------------------------- |
| Cloud-native           | Medium                         | High                                        |
| Approval complexity    | Lower to medium                | Medium to high                              |
| On-prem dependency     | Yes                            | No                                          |
| Vendor direct-to-Azure | No                             | Yes                                         |
| Operational complexity | Higher                         | Lower after setup                           |
| Long-term scalability  | Medium                         | High                                        |
| Security governance    | Uses existing controls         | Requires new Azure SFTP controls            |
| Time to implement      | Faster if mover/on-prem exists | Faster only if HNS/SFTP approvals are ready |
| Best use               | Short-term bridge              | Strategic target state                      |

## Recommended approach

Use a **two-step strategy**:

```text
Short term:
Use Mover if the vendor file path already exists on-prem and timelines are tight.

Long term:
Move to Azure Blob with HNS + SFTP enabled as the strategic cloud-native pattern.
```

## Recommendation wording for approval discussion

You can say:

> We have two options for receiving vendor files into Azure. The Mover option allows us to continue receiving files through the existing on-prem pattern and move them into Azure Blob, which may reduce immediate approval complexity but keeps on-prem dependency and adds operational hops. The HNS + SFTP option allows the vendor to securely send files directly to Azure Blob using native SFTP, which is the preferred long-term cloud-native pattern, but it requires approval because HNS changes the storage account behavior and introduces external SFTP access controls.
>
> Our recommendation is to use Mover as a short-term bridge only if timelines require it, and adopt a dedicated new Azure Storage Account with HNS and SFTP enabled as the target-state solution.
