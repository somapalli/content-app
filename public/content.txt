## 1. Business Use Cases

The enhanced transaction model handles seven real-world scenarios covering the complete range of payment lifecycle complexity encountered in a bank processing 2 million records per settlement window.

| ID | Merchant | Scenario | Description |
|---|---|---|---|
| SC-01 | Hyatt Regency | Hotel: Auth → Incremental Auth → Capture → Settlement | Guest checks in: $200 auth placed. Extends stay: +$150 incremental auth. Checks out: captures $310. Settles overnight. Classic hospitality lifecycle with incremental authorization. |
| SC-02 | Walgreens | Retail: Auth → Partial Void → Capture → Settlement | Customer buys pharmacy items, auth $85. Cashier removes one item ($15). Partial void reduces hold. Captures remaining $70 and settles same night. |
| SC-03 | Amazon | eCommerce: Multi-Capture → Multi-Settlement → Partial Refund | Order has 2 items shipped separately. Auth $500. Capture $200 (Item 1 ships Day 1). Capture $300 (Item 2 ships Day 3). Two separate settlements. Customer returns Item 2 → partial refund $300 against second settlement only. |
| SC-04 | Delta Airlines | Travel: Auth → Capture → Settlement → Chargeback Lost | Passenger buys $450 ticket. Settles. Files chargeback (never received service). Merchant contests and loses. Funds clawed back. Full dispute lifecycle. |
| SC-05 | Marriott Hotels | Hospitality: Multi-Settlement in Batches | Conference booking $2,000. Captures in three batches: $600 (room), $900 (conference hall), $500 (catering). Three separate settlement events tied to respective captures. |
| SC-06 | ABC Auto Repair | Over-Capture Exception + Adjustment | Auth $150. Capture attempt $200 blocked by trigger. Merchant gets incremental auth first. Manual positive adjustment $25 issued separately. |
| SC-07 | Hyatt Regency | Monthly Fee + Chargeback Won | Monthly gateway fee $49.95 applied. Dispute from March stay filed. Bank rules in merchant favour. Chargeback reversed, funds returned. |

### 1.1 Scenario SC-03 Deep Dive: Why Multi-Capture Matters

SC-03 (Amazon multi-shipment) is the most architecturally significant because it exposes every gap in a simple single-capture model:

- **Auth $500** establishes a single hold on the card.
- **Capture $200** (Item 1) does not release the entire hold — $300 must remain for Item 2.
- **Capture $300** (Item 2, 3 days later) resolves the remaining hold.
- **Settlement $200** runs that night for Item 1 only.
- **Settlement $300** runs 3 nights later when Item 2 ships.
- **Refund $300** goes against the **second settlement only** — not the transaction total.

> **⚠ Without Enhancement:**  
> A refund validation checking `SUM(settlements) - SUM(refunds)` sees `$500 settled - $300 refunded = $200 remaining` → allows the refund. But it **cannot prevent a second $300 refund** because the cumulative check still passes: `$500 - $300 - $300 = -$100` — which only fails on the second attempt.  
> The enhanced model tracks `refundable_balance` **per settlement event**, making each settlement individually accountable.

## 2. Data Model Architecture

The model is organized into six functional layers. Each layer has a single, clear responsibility.

| # | Layer | Tables | Responsibility | Change Frequency |
|---|---|---|---|---|
| 1 | Vocabulary (Lookup) | `lkp_transaction_type` `lkp_transaction_subtype` `lkp_transaction_status` `lkp_event_type` `lkp_valid_type_subtype` `lkp_status_transition` | Define what types, subtypes, statuses and events are legally allowed. Enforce valid combinations as database constraints. | Rare — driven by new network requirements or regulatory changes |
| 2 | Entity | `merchant` `merchant_account` | Identify the parties involved in payment processing. Anchor all transactions to a real business entity. | Low — merchant onboarding and account changes |
| 3 | Transaction (Intent) | `txn` | One row per business request. Records who, what, how much, and current lifecycle state. The anchor for all other data. | Moderate — status updated as lifecycle progresses |
| 4 | Event (History) | `txn_event` | Immutable append-only log of every state-changing occurrence. Source of truth for what happened. Enhanced with capture-level status tracking. | High — multiple events per transaction |
| 5 | Allocation (Money Movement) | `txn_allocation` | Explicit double-entry money trail. Every dollar's source and destination. Groups net to zero. | High — one group per event movement |
| 6 | Monitoring | `txn_monitor` `txn_exception` | Pre-aggregated CQRS read model for dashboards. Structured exception log for operational alerts. | Very High — updated on every event |

### 2.1 Key Design Principles

- **Immutability at the right level.** `txn_event` rows are never updated after insert. Corrections create new reversal events. Tamper-evident by design.
- **Running balances for O(1) enforcement.** Capture/refund validation reads a single column on the `txn` row, not an aggregation over event history. Critical for 2M record throughput.
- **Double-entry net-zero invariant.** Every allocation group must sum to zero. A single `GROUP BY` query confirms the entire ledger is balanced.
- **State machine as data.** Valid status transitions live in `lkp_status_transition`. Adding a new path requires an INSERT, not a code deployment.
- **Settlement-level balance tracking.** *(NEW v2)* Each settlement event tracks its own `refundable_balance`, enabling per-settlement refund validation.
- **Capture-level lifecycle tracking.** *(NEW v2)* Each capture event tracks its own `capture_status` and which settlement event resolved it.

## 3. Lookup Tables — The Vocabulary Layer

Lookup tables answer: *"What values are legally allowed in this system?"* Without them, free-text fields create inconsistency, indexes are ineffective, and application services drift apart.

---

### 3.1 `lkp_transaction_type`

**Purpose:** The top-level classification for every transaction. Every row in `txn` must reference exactly one type. The type determines which subtypes are valid, which statuses apply, and which state machine paths are available.

> **WHY this table exists:**  
> Without it, transaction type is stored as free-text VARCHAR2. Three teams use `"CC"`, `"CreditCard"`, and `"credit_card"` for the same thing. Adding a new type requires code changes in every service. With this table: type is FK-backed. New types need one INSERT. All services share the same vocabulary automatically.

#### Column Definitions

| Column | Type | Nullable | Purpose |
|---|---|---|---|
| `type_id` | `NUMBER(4) PK` | NOT NULL | System-generated identity. Referenced as FK by txn, lkp_transaction_subtype, lkp_transaction_status, lkp_valid_type_subtype, lkp_status_transition. |
| `type_code` | `VARCHAR2(30)` | NOT NULL | Short code used in application logic and API responses. UNIQUE constraint ensures no duplicate codes. |
| `type_name` | `VARCHAR2(100)` | NOT NULL | Human-readable display name for UIs and reports. |
| `description` | `VARCHAR2(500)` | NULL | Extended description including scope and business rules for this type. |
| `is_active` | `NUMBER(1)` | NOT NULL | Soft delete flag. Setting to 0 disables the type without losing historical references. |
| `created_at` | `TIMESTAMP` | NOT NULL | Audit timestamp for when the type was introduced into the system. |

#### DDL

```sql
CREATE TABLE lkp_transaction_type (
    type_id      NUMBER(4)     GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    type_code    VARCHAR2(30)  NOT NULL,
    type_name    VARCHAR2(100) NOT NULL,
    description  VARCHAR2(500),
    is_active    NUMBER(1)     DEFAULT 1 NOT NULL CHECK (is_active IN (0,1)),
    created_at   TIMESTAMP     DEFAULT SYSTIMESTAMP NOT NULL,
    CONSTRAINT uq_txn_type_code UNIQUE (type_code)
);
```

#### Sample Data

| type_id | type_code | type_name | Description |
|---|---|---|---|
| 1 | CC | Credit Card | Card-based payment transactions including auth, capture, void, refund. Most complex lifecycle. |
| 2 | FEE | Fee | Non-transactional fees charged to merchants: gateway, statement, PCI non-compliance. |
| 3 | ADJUSTMENT | Adjustment | Manual balance corrections by bank operators: over-fee credits, goodwill adjustments. |
| 4 | CHARGEBACK | Chargeback | Network-initiated cardholder disputes. Multi-step lifecycle: initiated, won, lost. |
| 5 | INSTALLMENT | Installment | Buy-now-pay-later plans. Partial captures on a recurring schedule against one auth. |

### 3.2 `lkp_transaction_subtype`

**Purpose:** Narrows the transaction type to a specific operation. Every subtype belongs to exactly one parent type via the `type_id` FK. Combined with the `lkp_valid_type_subtype` matrix FK on `txn`, this makes invalid type/subtype combinations physically impossible to insert.

> **WHY this table exists:**  
> A Credit Card transaction can be an Authorization, a Capture, a Void, or a Refund. These are fundamentally different operations with different amounts, different event chains, and different allocation patterns. Without subtypes, the transaction table cannot distinguish them. The subtype also serves as the key to the `lkp_valid_type_subtype` matrix, which prevents a Fee transaction from ever being given a `CC_AUTH` subtype at the database constraint level.

#### DDL

```sql
CREATE TABLE lkp_transaction_subtype (
    subtype_id    NUMBER(4)     GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    type_id       NUMBER(4)     NOT NULL,
    subtype_code  VARCHAR2(30)  NOT NULL,
    subtype_name  VARCHAR2(100) NOT NULL,
    description   VARCHAR2(500),
    is_active     NUMBER(1)     DEFAULT 1 NOT NULL CHECK (is_active IN (0,1)),
    created_at    TIMESTAMP     DEFAULT SYSTIMESTAMP NOT NULL,
    CONSTRAINT uq_txn_subtype_code UNIQUE (subtype_code),
    CONSTRAINT fk_subtype_type FOREIGN KEY (type_id)
        REFERENCES lkp_transaction_type (type_id)
);
```

#### Sample Data

| Parent Type | subtype_code | subtype_name | When Used |
|---|---|---|---|
| CC | CC_AUTH | Authorization | Merchant requests hold on card. No money moves. First step in CC lifecycle. |
| CC | CC_INCR_AUTH | Incremental Auth | Hotel/car rental adds to existing hold without releasing original. |
| CC | CC_CAPTURE | Capture | Merchant claims authorized funds. Money enters clearing. |
| CC | CC_PARTIAL_VOID | Partial Void | Reduces existing auth hold. Item removed from order. |
| CC | CC_FULL_VOID | Full Void | Cancels entire authorization before capture. |
| CC | CC_REFUND | Refund | Returns settled funds to cardholder. Child transaction of original. |
| FEE | FEE_MONTHLY | Monthly Fee | Recurring gateway usage charge. |
| FEE | FEE_STATEMENT | Statement Fee | Paper statement surcharge. |
| ADJUSTMENT | ADJ_POSITIVE | Positive Adjustment | Bank credits merchant — fee correction. |
| ADJUSTMENT | ADJ_NEGATIVE | Negative Adjustment | Bank debits merchant — returned item. |
| CHARGEBACK | CB_INITIATED | Chargeback Initiated | Dispute filed by cardholder. |
| CHARGEBACK | CB_REVERSED | Chargeback Reversed | Merchant won — funds returned. |

### 3.3 `lkp_transaction_status`

**Purpose:** Defines every possible lifecycle state a transaction can be in, scoped to its parent type. The `is_terminal` flag marks states from which no further transitions are permitted.

> **WHY `is_terminal` exists:**  
> Settlement batch job needs to answer: *"Can I still process this transaction?"*  
> Without `is_terminal`, it must hardcode: `IF status IN ('CC_VOIDED','CC_REFUNDED',...)`. When you add a new terminal status, every batch job must be updated.  
> With `is_terminal = 1`: `SELECT * FROM txn WHERE status.is_terminal = 0` — adding a new terminal status requires one UPDATE on the lookup row. Every batch job automatically excludes it without code changes.

#### DDL

```sql
CREATE TABLE lkp_transaction_status (
    status_id    NUMBER(4)     GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    type_id      NUMBER(4)     NOT NULL,
    status_code  VARCHAR2(30)  NOT NULL,
    status_name  VARCHAR2(100) NOT NULL,
    is_terminal  NUMBER(1)     DEFAULT 0 NOT NULL CHECK (is_terminal IN (0,1)),
    is_active    NUMBER(1)     DEFAULT 1 NOT NULL CHECK (is_active IN (0,1)),
    created_at   TIMESTAMP     DEFAULT SYSTIMESTAMP NOT NULL,
    CONSTRAINT uq_txn_status_code UNIQUE (status_code),
    CONSTRAINT fk_status_type FOREIGN KEY (type_id)
        REFERENCES lkp_transaction_type (type_id)
);
```

#### Sample Data

| Type | status_code | Terminal? | Business Meaning |
|---|---|---|---|
| CC | CC_AUTHORIZED | NO | Auth hold placed. Card reserved but no money moved yet. Can capture or void. |
| CC | CC_PARTIALLY_CAPTURED | NO | **NEW** — Some captures done, more pending. Mixed lifecycle state. |
| CC | CC_CAPTURED | NO | All captures done. Awaiting settlement. |
| CC | CC_PARTIALLY_SETTLED | NO | **NEW** — Some settlements done, more pending. |
| CC | CC_SETTLED | NO | Fully settled. Funds transferred. Can be refunded or chargedback. |
| CC | CC_VOIDED | **YES** | Auth cancelled before capture. No money moved. Final state. |
| CC | CC_REFUNDED | **YES** | Fully refunded. All settled funds returned. Final state. |
| CC | CC_CHARGEDBACK | **YES** | Dispute resolved. Final state regardless of outcome. |
| FEE | FEE_PENDING | NO | Fee calculated, awaiting application. |
| FEE | FEE_APPLIED | **YES** | Fee debited. Cannot be reversed — create ADJ_POSITIVE instead. |
| CHARGEBACK | CB_INITIATED | NO | Dispute open. Merchant must respond within deadline. |
| CHARGEBACK | CB_WON | **YES** | Ruled for merchant. Funds retained. Final. |
| CHARGEBACK | CB_LOST | **YES** | Ruled for cardholder. Funds clawed back. Final. |

### 3.4 `lkp_event_type`

**Purpose:** Defines every specific thing that can happen in a transaction's lifecycle. Unlike transaction subtypes, event types are **not scoped to a parent transaction type** — the same `SETTLEMENT` event type appears on CC, FEE, and ADJUSTMENT transactions. The `affects_balance` flag tells the running balance trigger which events should update balance columns on the `txn` row.

#### DDL

```sql
CREATE TABLE lkp_event_type (
    event_type_id    NUMBER(4)     GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    event_type_code  VARCHAR2(30)  NOT NULL,
    event_type_name  VARCHAR2(100) NOT NULL,
    affects_balance  NUMBER(1)     DEFAULT 1 NOT NULL CHECK (affects_balance IN (0,1)),
    description      VARCHAR2(500),
    is_active        NUMBER(1)     DEFAULT 1 NOT NULL CHECK (is_active IN (0,1)),
    created_at       TIMESTAMP     DEFAULT SYSTIMESTAMP NOT NULL,
    CONSTRAINT uq_evt_type_code UNIQUE (event_type_code)
);
```

#### Sample Data

| event_type_code | event_type_name | Balance Impact | When It Occurs |
|---|---|---|---|
| AUTH | Authorization | `+authorized_balance` | Merchant sends auth request. Card network approves hold. |
| INCR_AUTH | Incremental Auth | `+authorized_balance` | Hotel/rental adds to existing hold. |
| CAPTURE | Capture | `+captured_balance` | Merchant claims funds. Captured balance increases. |
| PARTIAL_VOID | Partial Void | `-authorized_balance` | Item removed. Authorized balance decreases. |
| FULL_VOID | Full Void | `authorized_balance = 0` | Entire auth cancelled. |
| SETTLEMENT | Settlement | `+settled_balance` | Nightly batch. Funds physically transferred to merchant bank. |
| REFUND | Refund | `+refunded_balance` | Funds returned from merchant to cardholder post-settlement. |
| FEE_APPLIED | Fee Applied | `+settled_balance (fee)` | Monthly or statement fee debited from merchant account. |

| STATUS_CHECK | Status Check | None | Ping or inquiry. No financial impact. |

### 3.5 `lkp_valid_type_subtype`

**Purpose:** The **enforcement table**, not just a reference table. Its composite primary key `(type_id, subtype_id)` is referenced by a **composite foreign key on `txn`** — meaning the database engine itself validates the type/subtype pair on every INSERT. No application code, trigger, or explicit check is needed.

> **HOW the three-constraint chain works:**
>
> The `txn` table has **three foreign key constraints** working together:
>
> - **Constraint 1:** `fk_txn_type` → validates `type_id` exists in `lkp_transaction_type`
> - **Constraint 2:** `fk_txn_subtype` → validates `subtype_id` exists in `lkp_transaction_subtype`
> - **Constraint 3:** `fk_txn_vts` → validates `(type_id, subtype_id)` **pair** exists in `lkp_valid_type_subtype`
>
> Constraints 1 and 2 each validate one column independently. Constraint 3 validates the **combination**.  
> `type = FEE` with `subtype = CC_AUTH` passes constraints 1 and 2 (both exist individually) but **fails constraint 3** (the pair `(FEE, CC_AUTH)` is not in the matrix). INSERT is rejected.

#### DDL

```sql
CREATE TABLE lkp_valid_type_subtype (
    type_id     NUMBER(4)  NOT NULL,
    subtype_id  NUMBER(4)  NOT NULL,
    is_active   NUMBER(1)  DEFAULT 1 NOT NULL CHECK (is_active IN (0,1)),
    CONSTRAINT pk_valid_type_subtype PRIMARY KEY (type_id, subtype_id),
    CONSTRAINT fk_vts_type    FOREIGN KEY (type_id)
        REFERENCES lkp_transaction_type (type_id),
    CONSTRAINT fk_vts_subtype FOREIGN KEY (subtype_id)
        REFERENCES lkp_transaction_subtype (subtype_id)
);

-- Composite FK on txn referencing this matrix:
CONSTRAINT fk_txn_vts FOREIGN KEY (type_id, subtype_id)
    REFERENCES lkp_valid_type_subtype (type_id, subtype_id)
```

The **absence of a row is the enforcement** — the matrix contains only valid pairs. Any unlisted combination is physically impossible.

### 3.6 `lkp_status_transition`

**Purpose:** Defines the complete state machine for each transaction type as database rows. Each row represents one allowed transition: from one status to another, for a specific transaction type. A trigger on `txn` queries this table on every status change and rejects the update if no matching active row exists.

> **WHY state machine as data (not code):**  
> Six services handle status updates: gateway, settlement engine, refund processor, chargeback handler, fee engine, and operator console.  
> **If state machine lives in code:** Six implementations. Each can drift. Adding one new status path requires finding and updating all six.  
> **If state machine lives in data:** One trigger queries one table. All six services get identical enforcement automatically. Adding a new path = one INSERT, zero code changes.  
> The `is_active` flag on each row allows disabling a path (regulatory change) without deleting historical evidence that the path once existed.

#### DDL

```sql
CREATE TABLE lkp_status_transition (
    transition_id    NUMBER(6)  GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    type_id          NUMBER(4)  NOT NULL,
    from_status_id   NUMBER(4),              -- NULL = initial state (brand new transaction)
    to_status_id     NUMBER(4)  NOT NULL,
    is_active        NUMBER(1)  DEFAULT 1 NOT NULL CHECK (is_active IN (0,1)),
    CONSTRAINT uq_status_transition UNIQUE (type_id, from_status_id, to_status_id),
    CONSTRAINT fk_st_type  FOREIGN KEY (type_id)
        REFERENCES lkp_transaction_type (type_id),
    CONSTRAINT fk_st_from  FOREIGN KEY (from_status_id)
        REFERENCES lkp_transaction_status (status_id),
    CONSTRAINT fk_st_to    FOREIGN KEY (to_status_id)
        REFERENCES lkp_transaction_status (status_id)
);

-- Enforcing trigger
CREATE OR REPLACE TRIGGER trg_validate_status_transition
BEFORE UPDATE OF status_id ON txn
FOR EACH ROW
DECLARE
    v_exists NUMBER := 0;
BEGIN
    IF :OLD.status_id = :NEW.status_id THEN RETURN; END IF;
    SELECT COUNT(*) INTO v_exists
    FROM lkp_status_transition
    WHERE type_id = :NEW.type_id
      AND NVL(from_status_id, -1) = NVL(:OLD.status_id, -1)
      AND to_status_id = :NEW.status_id
      AND is_active = 1;
    IF v_exists = 0 THEN
        RAISE_APPLICATION_ERROR(-20005,
            'INVALID STATUS TRANSITION for txn ' || :NEW.transaction_id);
    END IF;
END;
```

#### Credit Card State Machine (Transition Rows)

```
-- NULL from = initial state
(CC, NULL,                    CC_AUTHORIZED)          -- new txn starts AUTHORIZED
(CC, CC_AUTHORIZED,           CC_PARTIALLY_CAPTURED)  -- first of multiple captures
(CC, CC_AUTHORIZED,           CC_CAPTURED)            -- single full capture
(CC, CC_AUTHORIZED,           CC_VOIDED)              -- cancel before any capture
(CC, CC_PARTIALLY_CAPTURED,   CC_CAPTURED)            -- last capture completes
(CC, CC_CAPTURED,             CC_PARTIALLY_SETTLED)   -- first of multiple settlements
(CC, CC_CAPTURED,             CC_SETTLED)             -- single full settlement
(CC, CC_PARTIALLY_SETTLED,    CC_SETTLED)             -- last settlement completes
(CC, CC_SETTLED,              CC_REFUNDED)            -- post-settlement refund
(CC, CC_SETTLED,              CC_CHARGEDBACK)         -- network dispute

-- These rows DO NOT EXIST — absence is the enforcement:
-- CC_AUTHORIZED  → CC_SETTLED      (cannot skip capture)
-- CC_VOIDED      → anything        (terminal: no outbound rows)
-- CC_REFUNDED    → anything        (terminal: no outbound rows)
```

## 5. Transaction Table (`txn`)

**Layer:** Transaction Intent  
**Purpose:** Records what the merchant requested and what the current outcome is. One row per business request. Every event, allocation, and monitoring record exists to serve or describe a transaction row.

### 5.1 The Three-Amount Model

> **NOTE:**  
> A single `amount` column cannot describe a transaction that was originally authorized for $200, had an incremental auth to $350, was captured for $310, and settled for $310.
>
> | Column | Set At | Business Meaning |
> |---|---|---|
> | `requested_amount` | INSERT | What was originally submitted. Never changes. |
> | `approved_amount` | After AUTH / INCR_AUTH | What the card network authorized. |
> | `settled_amount` | After SETTLEMENT | What actually moved to merchant bank. |
>
> All three must coexist. Loss of any one makes reconciliation impossible.

### 5.2 Running Balance Columns

> **WHY running balances (not aggregation):**  
> Capture validation: *"is the capture amount ≤ authorized balance?"*  
> **Naive approach:** `SUM` all AUTH + INCR_AUTH events where `transaction_id = X` → at 2M transactions × 3–5 events = 10M rows aggregated constantly under concurrent load.  
> **Running balance approach:** `UPDATE txn SET current_authorized_balance += :event_amount` — the trigger maintains the column. Capture validation reads ONE indexed row. **O(1).**  
> These columns are derived from events. Events remain the source of truth. A nightly reconciliation verifies columns match event sums.

### 5.3 Column Definitions

| Column | Type | Nullable | Purpose |
|---|---|---|---|

## 6. Event Table (`txn_event`)

**Layer:** Event History  
**Purpose:** Every state-changing occurrence in a transaction's life is recorded as an immutable event row. The event table is the **source of truth** — the transaction's current state can be fully reconstructed from its event history at any point in time.

### 6.1 Key v2 Enhancements — Capture-Level Tracking

> **NOTE:**  
> The core gap from v1: events knew WHAT happened but not WHERE in the capture/settlement chain.  
>
> **Problem:** With multiple captures and settlements, which capture does settlement X resolve? Which settlement events are still pending? How much of settlement Y is refundable?  
>
> **Solution — five new columns on `txn_event`:**
>
> | Column | Purpose |
> |---|---|
> | `capture_status` | For CAPTURE events only: `PENDING_SETTLEMENT`, `SETTLED`, `REFUNDED` |
> | `settlement_event_id` | FK: which SETTLEMENT event resolved this CAPTURE |
> | `refundable_balance` | For SETTLEMENT events: how much can still be refunded |
> | `source_capture_id` | FK: which CAPTURE does this SETTLEMENT resolve |
> | `total_sourced_amount` | How much of this event has been used as SOURCE in allocation groups — prevents over-sourcing |

### 6.2 Column Definitions

| Column | Type | Nullable | Purpose |
|---|---|---|---|
| `event_id` | `VARCHAR2(36) PK` | NOT NULL | UUID primary key. Referenced by allocation table and reversal chain. |
| `transaction_id` | `VARCHAR2(36) FK` | NOT NULL | References parent `txn` row. Every event belongs to exactly one transaction. |
| `event_type_id` | `NUMBER(4) FK` | NOT NULL | References `lkp_event_type`. What kind of thing happened. |
| `sequence_number` | `NUMBER(6)` | NOT NULL | Integer, unique per transaction. Detects out-of-order network delivery. |
| `amount` | `NUMBER(18,2)` | NOT NULL | Signed amount for this event. Negative for voids and refunds. |

```sql
-- Immutability trigger
CREATE OR REPLACE TRIGGER trg_event_immutability
BEFORE UPDATE OR DELETE ON txn_event FOR EACH ROW
BEGIN
    RAISE_APPLICATION_ERROR(-20010,
        'Event ' || :OLD.event_id || ' is immutable. Create a reversal event instead.');
END;

-- Capture validation trigger
CREATE OR REPLACE TRIGGER trg_validate_capture_amount
BEFORE INSERT ON txn_event FOR EACH ROW
DECLARE
    v_code     VARCHAR2(30);
    v_auth_bal NUMBER;
BEGIN
    SELECT et.event_type_code INTO v_code
    FROM lkp_event_type et WHERE et.event_type_id = :NEW.event_type_id;
    IF v_code = 'CAPTURE' THEN
        SELECT current_authorized_balance INTO v_auth_bal
        FROM txn WHERE transaction_id = :NEW.transaction_id FOR UPDATE;
        IF :NEW.amount > v_auth_bal THEN
            RAISE_APPLICATION_ERROR(-20004,
                'CAPTURE ' || :NEW.amount || ' exceeds authorized balance ' || v_auth_bal);
        END IF;
    END IF;
END;

-- Running balance + capture tracking trigger
CREATE OR REPLACE TRIGGER trg_maintain_running_balance
AFTER INSERT ON txn_event FOR EACH ROW
DECLARE v_code VARCHAR2(30);
BEGIN
    SELECT event_type_code INTO v_code
    FROM lkp_event_type WHERE event_type_id = :NEW.event_type_id;
    UPDATE txn SET
        current_authorized_balance = current_authorized_balance +
            CASE WHEN v_code IN ('AUTH','INCR_AUTH') THEN :NEW.amount
                 WHEN v_code = 'PARTIAL_VOID'        THEN :NEW.amount ELSE 0 END,
        current_captured_balance   = current_captured_balance +
            CASE WHEN v_code = 'CAPTURE'     THEN :NEW.amount ELSE 0 END,
        current_settled_balance    = current_settled_balance +
            CASE WHEN v_code = 'SETTLEMENT'  THEN :NEW.amount ELSE 0 END,
        current_refunded_balance   = current_refunded_balance +
            CASE WHEN v_code = 'REFUND'      THEN ABS(:NEW.amount) ELSE 0 END,
        pending_settlement_amount  = current_captured_balance - current_settled_balance,
        capture_count    = capture_count    + CASE WHEN v_code='CAPTURE'    THEN 1 ELSE 0 END,
        settlement_count = settlement_count + CASE WHEN v_code='SETTLEMENT' THEN 1 ELSE 0 END,
        updated_at = SYSTIMESTAMP
    WHERE transaction_id = :NEW.transaction_id;
    -- Initialize capture_status on new CAPTURE events
    IF v_code = 'CAPTURE' THEN
        UPDATE txn_event SET capture_status = 'PENDING_SETTLEMENT'
        WHERE event_id = :NEW.event_id;
    END IF;
    -- Initialize refundable_balance on new SETTLEMENT events
    IF v_code = 'SETTLEMENT' THEN
        UPDATE txn_event SET refundable_balance = :NEW.amount
        WHERE event_id = :NEW.event_id;
    END IF;
END;
```

## 7. Allocation Table (`txn_allocation`)

**Layer:** Money Movement  
**Purpose:** Records exactly where every dollar came from and where it went. An event says *"a capture of $310 happened"* — the allocation says *"that $310 came from the auth hold established by Event E1 and went into the clearing account."*

### 7.1 The Double-Entry Invariant

> **KEY:**  
> For every `allocation_group_id`:  
> ```
> SUM(amount WHERE role = 'SOURCE') + SUM(amount WHERE role = 'TARGET') = 0
> ```
> `SOURCE` lines carry **negative** amounts (money leaving).  
> `TARGET` lines carry **positive** amounts (money arriving).  
>
> A single `GROUP BY allocation_group_id` across all allocations confirms the entire ledger is balanced.  
> Any non-zero result pinpoints the exact money movement that is out of balance.

### 7.2 Over-Sourcing Prevention

> **WHY this trigger exists:**  
> Event E2 holds an authorization of $350. Two captures draw from E2:  
> - `AG-3: E2 SOURCE -$200, E3 TARGET +$200` → NET $0 ✓  
> - `AG-4: E2 SOURCE -$300, E4 TARGET +$300` → NET $0 ✓  
> - `AG-5: E2 SOURCE -$200, E5 TARGET +$200` → NET $0 ✓ ← **$700 sourced from $350 hold!**  
>
> Each group individually nets to zero. The over-sourcing is invisible to the net-zero check.  
>
> **Solution:** `total_sourced_amount` on `txn_event` tracks how much of each event has been used as a SOURCE. The allocation INSERT trigger rejects any sourcing that would cause `total_sourced_amount` to exceed the event's own amount.

### 7.3 DDL

```sql
CREATE TABLE txn_allocation (
    allocation_id        NUMBER(12)    GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    allocation_group_id  VARCHAR2(36)  NOT NULL,
    event_id             VARCHAR2(36)  NOT NULL,
    role                 VARCHAR2(10)  NOT NULL CHECK (role IN ('SOURCE','TARGET')),
    flow_type            VARCHAR2(30)  NOT NULL,
    amount               NUMBER(18,2)  NOT NULL CHECK (amount <> 0),
    ledger_entry_id      VARCHAR2(36),
    allocated_at         TIMESTAMP     DEFAULT SYSTIMESTAMP NOT NULL,
    -- Enforce sign matches role
    CONSTRAINT chk_alloc_sign CHECK (
        (role = 'SOURCE' AND amount < 0) OR (role = 'TARGET' AND amount > 0)
    ),
    CONSTRAINT fk_alloc_event FOREIGN KEY (event_id) REFERENCES txn_event (event_id)
);

-- Over-sourcing prevention trigger
CREATE OR REPLACE TRIGGER trg_validate_allocation_source
BEFORE INSERT ON txn_allocation FOR EACH ROW
DECLARE
    v_event_amount    NUMBER(18,2);
    v_already_sourced NUMBER(18,2);
BEGIN
    IF :NEW.role = 'SOURCE' THEN
        SELECT amount, total_sourced_amount
        INTO v_event_amount, v_already_sourced
        FROM txn_event WHERE event_id = :NEW.event_id FOR UPDATE;
        IF (v_already_sourced + ABS(:NEW.amount)) > v_event_amount THEN
            RAISE_APPLICATION_ERROR(-20007,
                'Over-sourcing event ' || :NEW.event_id ||
                '. Capacity: ' || v_event_amount ||
                ', Already sourced: ' || v_already_sourced ||
                ', Attempted: ' || ABS(:NEW.amount));
        END IF;
        UPDATE txn_event
        SET total_sourced_amount = total_sourced_amount + ABS(:NEW.amount)
        WHERE event_id = :NEW.event_id;
    END IF;
END;

-- Net-zero reconciliation query (run daily)
SELECT
    allocation_group_id,
    SUM(amount) AS net_balance,
    CASE WHEN SUM(amount) = 0 THEN 'BALANCED' ELSE 'IMBALANCED' END AS status
FROM txn_allocation
GROUP BY allocation_group_id
HAVING SUM(amount) <> 0;  -- returns only broken groups
```

## 8. Monitoring Tables

### 8.1 `txn_monitor`

**Layer:** Monitoring (CQRS Read Model)  
**Purpose:** A pre-aggregated summary table maintained by triggers. Answers operational questions without joining the normalized write model. Implements **CQRS (Command Query Responsibility Segregation)** within Oracle.

> **WHY a separate table (not a view or materialized view):**  
>
> - **Direct query of normalized model:** `JOIN txn + txn_event + txn_allocation + GROUP BY` under 2M record load = lock contention between analytical reads and transactional writes.  
> - **Oracle Materialized View with `REFRESH ON COMMIT`:** Refresh adds overhead to every write. Deferred refresh = stale data. Cannot hold computed columns like `anomaly_flags`.  
> - **Maintained summary table (chosen):** Updated by `AFTER INSERT` trigger on `txn_event` in the same commit. **No stale data window.** Dashboard reads ONE indexed row.

```sql
CREATE TABLE txn_monitor (
    transaction_id              VARCHAR2(36)  PRIMARY KEY,
    merchant_id                 VARCHAR2(20)  NOT NULL,
    current_status_code         VARCHAR2(30)  NOT NULL,

    -- Pre-aggregated financial totals
    total_authorized            NUMBER(18,2)  DEFAULT 0,
    total_captured              NUMBER(18,2)  DEFAULT 0,
    total_settled               NUMBER(18,2)  DEFAULT 0,
    total_refunded              NUMBER(18,2)  DEFAULT 0,
    total_chargedback           NUMBER(18,2)  DEFAULT 0,
    net_position                NUMBER(18,2)  DEFAULT 0,
    pending_settlement_amount   NUMBER(18,2)  DEFAULT 0,
    -- Multi-capture/settlement counters

    -- Flags
```

### 8.2 `txn_exception`

**Layer:** Monitoring (Exception Log)  
**Purpose:** Structured exception log separating **detection** (log a row) from **prevention** (block the operation). Some violations should be flagged for human review rather than automatically rejected — e.g. a capture 5% over auth in a hotel scenario.

|---|---|---|---|
| `CRITICAL` | Duplicate authorization detected | Auto-reject + real-time alert | Immediate |
| `CRITICAL` | Chargeback received — deadline in 24h | Ops team paged | Immediate |
| `WARNING` | Capture exceeds auth by < 20% | Human review queue | 4 hours |
| `WARNING` | Settlement delayed > 48 hours | Ops team notified | 4 hours |
| `INFO` | Incremental auth on non-hotel MCC | Logged only | 24 hours |

## 9. Sample Data

### SC-01: Hyatt Regency — Incremental Auth → Capture → Settlement

**Business Story:** Guest checks in March 15. Initial auth $200. Extends stay — incremental auth +$150 (total hold $350). Checks out March 17 — captures $310. Settles overnight at $310.

```sql
-- Transaction header
INSERT INTO txn (
    transaction_id, merchant_id, account_id, terminal_id,
    type_id, subtype_id, status_id,
    requested_amount, approved_amount, settled_amount, currency_code,
    current_authorized_balance, current_captured_balance, current_settled_balance,

    idempotency_key, external_ref_id, entry_mode, channel,
```

### Verification Queries

```sql
-- 1. Full event timeline for any transaction
SELECT
    e.sequence_number,
    et.event_type_code,
    e.amount,
    e.capture_status,
    e.refundable_balance,
    e.event_status,
    e.network_timestamp,
    e.initiated_by
FROM txn_event e
JOIN lkp_event_type et ON e.event_type_id = et.event_type_id
WHERE e.transaction_id = :txn_id
ORDER BY e.sequence_number;

-- 2. Net-zero check across entire ledger (0 rows = fully balanced)
SELECT
    allocation_group_id,
    SUM(amount) AS net,
    CASE WHEN SUM(amount) = 0 THEN 'BALANCED' ELSE 'IMBALANCED' END AS status
FROM txn_allocation
GROUP BY allocation_group_id
HAVING SUM(amount) <> 0;

-- 3. Captures pending settlement across all merchants
SELECT
    t.merchant_id,
    m.merchant_name,
    e.event_id,
    e.amount,
    ROUND((SYSTIMESTAMP - e.received_at) * 1440, 1) AS pending_minutes
FROM txn_event e
JOIN lkp_event_type et ON e.event_type_id = et.event_type_id
JOIN txn t             ON e.transaction_id = t.transaction_id
JOIN merchant m        ON t.merchant_id    = m.merchant_id
WHERE et.event_type_code = 'CAPTURE'
  AND e.capture_status   = 'PENDING_SETTLEMENT'
ORDER BY pending_minutes DESC;

-- 4. Per-settlement refundable balance
SELECT
    e.event_id,
    e.amount          AS settled,
    e.refundable_balance,
    (e.amount - e.refundable_balance) AS already_refunded,
    src.event_id      AS resolved_capture
FROM txn_event e
JOIN lkp_event_type et ON e.event_type_id = et.event_type_id
LEFT JOIN txn_event src ON e.source_capture_id = src.event_id
WHERE e.transaction_id = :txn_id
  AND et.event_type_code = 'SETTLEMENT'
ORDER BY e.sequence_number;

-- 5. Open exceptions by severity
SELECT
    ex.exception_code,
    ex.exception_severity,
    m.merchant_name,
    t.transaction_id,
    ex.exception_message,
    ROUND((SYSTIMESTAMP - ex.detected_at) * 1440, 1) AS age_minutes
FROM txn_exception ex
JOIN txn t      ON ex.transaction_id = t.transaction_id
JOIN merchant m ON t.merchant_id     = m.merchant_id
WHERE ex.is_resolved = 0
ORDER BY ex.exception_severity DESC, ex.detected_at;

-- 6. Parent-child full lifecycle chain (hierarchical query)
SELECT
    LEVEL,
    t.transaction_id,
    st.subtype_name,
    s.status_name,
    t.requested_amount,
    LPAD(' ', LEVEL * 4) || t.transaction_id AS tree
FROM txn t
JOIN lkp_transaction_subtype st ON t.subtype_id = st.subtype_id
JOIN lkp_transaction_status  s  ON t.status_id  = s.status_id
START WITH t.parent_transaction_id IS NULL
       AND t.merchant_id = :merchant_id
CONNECT BY PRIOR t.transaction_id = t.parent_transaction_id;

-- 7. Monitoring dashboard — transactions with anomalies
SELECT
    m.merchant_name,
    mon.transaction_id,
    mon.current_status_code,
    mon.sub_status,
    mon.total_captured,
    mon.total_settled,
    mon.pending_settlement_amount,
    mon.total_refunded,
    mon.total_chargedback,
    mon.anomaly_flags
FROM txn_monitor mon
JOIN merchant m ON mon.merchant_id = m.merchant_id
WHERE mon.anomaly_flags IS NOT NULL
ORDER BY m.merchant_name;
```

## 10. Design Summary

### Table Reference

| Table | Layer | Enforced By | Why It Exists |
|---|---|---|---|
| `lkp_transaction_type` | Vocabulary | UNIQUE + FK | Controls what types of transactions can exist. New types need one INSERT, not code changes. |
| `lkp_transaction_subtype` | Vocabulary | FK + matrix | Scopes operations to their parent type. Prevents CC subtypes on FEE transactions. |
| `lkp_transaction_status` | Vocabulary | FK + `is_terminal` | Defines lifecycle states per type. `is_terminal` drives all batch job exclusion logic. |
| `lkp_event_type` | Vocabulary | UNIQUE + FK | Defines what can happen. `affects_balance` controls running balance updates. |
| `lkp_valid_type_subtype` | Enforcement | Composite FK on `txn` | Makes invalid type/subtype pairs physically impossible. No trigger needed. |
| `lkp_status_transition` | State Machine | Trigger + data rows | All status transition rules in data. Adding paths needs no code deployment. |
| `merchant` | Entity | FK anchor | Real-world entity. Enables per-merchant monitoring, fees, and risk scoring. |
| `merchant_account` | Entity | FK | Routes settlement and fee debits to correct accounts per merchant. |
| `txn` | Transaction | Multi-constraint | Business intent anchor. Three-amount model. Running balances for O(1) validation. |
| `txn_event` | History | Immutability trigger | Append-only event log. Source of truth. Capture-level lifecycle tracking *(NEW v2)*. |
| `txn_allocation` | Money Movement | Net-zero + over-source trigger | Explicit double-entry trail. Single `GROUP BY` confirms entire ledger balance. |
| `txn_monitor` | Read Model | AFTER INSERT trigger | Pre-aggregated CQRS read side. Dashboard queries need no joins to normalized tables. |
| `txn_exception` | Exception Log | Application + trigger | Structured detection log. Graduated response: flag vs block. Full resolution audit trail. |

### v2 Enhancement Summary

| Enhancement | Columns Added | Problem Solved |
|---|---|---|
| Capture-level lifecycle tracking | `txn_event.capture_status` `txn_event.settlement_event_id` | Which captures are still pending settlement? Which settlement resolved each capture? |
| Settlement-level refund tracking | `txn_event.refundable_balance` `txn_event.source_capture_id` | Prevents over-refunding a specific settlement while cumulative total still permits it. |
| Allocation over-sourcing prevention | `txn_event.total_sourced_amount` + `trg_validate_allocation_source` | Prevents an auth hold from funding more captures than its value allows. |
| Mixed lifecycle sub-status | `txn.sub_status` `txn.pending_settlement_amount` | Expresses `PARTIAL_CAPTURED`/`PARTIAL_SETTLED` states a single `status_id` cannot represent. |
| Multi-event counters | `txn.capture_count` `txn.settlement_count` `txn.refund_count` | Fast query: how many captures/settlements on this transaction without scanning events. |
| New CC status states | `CC_PARTIALLY_CAPTURED` `CC_PARTIALLY_SETTLED` | Correctly models transactions mid-lifecycle with some but not all captures/settlements done. |
